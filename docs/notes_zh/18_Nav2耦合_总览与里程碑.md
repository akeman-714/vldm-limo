# 18 · VLFM 高层 + Nav2 驾驶（方案 B）总览与里程碑

> 本系列（18–24）是「VLFM 当大脑、Nav2 当司机、VLFM 订阅 LiDAR `/map`」这套耦合的**执行计划**。
> 续接 15/16/17（Limo Gazebo 接入）。每篇都给：规划 → 详细步骤 → 接口 → 注意点 → 可能问题+解法 → 验收。

## 0. 一句话架构

```
RTAB-Map(LiDAR+RGBD SLAM)
  ├─ TF: map→odom→base_link        （全局位姿）
  └─ /map  nav_msgs/OccupancyGrid  （360° LiDAR 融合占据图）
        │                  │
        ▼                  ▼
VLFM 大脑节点(你写)        Nav2(司机)
  订阅 RGBD/CameraInfo/TF/map        global+local planner + costmap(/scan)
  → value map(SigLIP2) + object map(YOLO/GDINO)
  → /map 注入 ObstacleMap → frontier
  → decide_goal() 输出 (x,y,yaw) @map ──► NavigateToPose ──► /cmd_vel ──► Limo
  ← on_goal_reached() ← Nav2 action result(SUCCEEDED/FAILED)
```

**两条唯一性铁律**：`/cmd_vel` 只有 Nav2 写；目标流只有 VLFM 写。其它 topic（相机/TF/`/map`）都是只读广播，VLFM 与 Nav2 同时订阅互不干扰。

## 1. 核心设计决定（为什么这么做）

| 决定 | 理由 | 影响面 |
|---|---|---|
| **episodic 米 ≡ map 帧米** | VLFM 内部所有图以 (x,y) 米为公共货币；统一成 map 帧后 frontier/物体坐标直接喂 Nav2，无需翻转 | 全局，见 19 |
| **删掉 Habitat 的 y 取反** | `habitat_policies.py:199` 的 `[x,-y]` 只是 Habitat GPS 把西设负；ROS TF 本就是右手系 | 19、21 |
| **障碍图改吃 `/map`（方案 B）** | 单前向相机视野窄→探索慢；用 360° LiDAR 融合图，与 Nav2 同源 | 20 |
| **绕过 PointNav / 退役 A\*** | Nav2 全局规划+局部避障接管驾驶；VLFM 的 ObstacleMap 退化成纯 frontier 探测器 | 22、23 |
| **决策层改进全保留** | 记忆/二次确认/reject 都在共享基类 `base_objectnav_policy.py`，是 goal 层逻辑 | 23 |
| **到达事件从 PointNav 上提到 Nav2 result** | `_called_stop` 原在 `_pointnav` 内部设；Nav2 驾驶后由 action result 宣告到达 | 23 |

## 3. 里程碑（Gate）—— 每个 Gate 自带验收，未过不进下一个

| Gate | 内容 | 通过标准（验收） |
|---|---|---|
| **G0 帧验证** | M1+M2 完成；`decide_goal` 只**打印** goal 不真发 | RViz 中 frontier 落在 `/map` 的「已知↔未知」边界上；打印的 (x,y) 与 RViz 量取一致（±1 格）；机器人朝 frontier 方向合理、**不反向** |
| **G1 接 Nav2（保守）** | M4 的 `goToPose` 闭环；发一个→等 `getResult`→再选 | 机器人能开到打印过的 frontier；`getResult()==SUCCEEDED`；中途不撞（Nav2 避障生效） |
| **G2 抢占+物体+到达接缝** | M5 抢占 + M4 `on_goal_reached`（verify/remember/reject） | 检测到目标物体→立刻抢占去物体；到点触发二次确认；确认失败→reject 拉黑并转探索；确认成功→写记忆 |
| **G3 调参收敛** | size/越界、`occ_thresh`、`yaw_goal_tolerance`、抢占阈值 | 完整一局：探索→发现→确认→停；无越界崩溃、无原地横跳、无对墙卡死 |

## 4. 全局配置：开什么、退什么

> 这些是**运行期开关**，集中在此，各模块文档只引用不重复。

**退役（方案 B 不再用）**
- `VLFM_GLOBAL_NAV`（A\*）—— 不设/设 0。Nav2 全局规划替代。
- PointNav 权重 —— 仍会被 `__init__` 加载（见 22 注意点），但**不参与驾驶**。
- `VLFM_PERSIST_MAP_PATH`（VLFM 自存障碍图）—— 冗余，RTAB-Map 自己持久化 `/map`。

**保留并照常开**
- `VLFM_OBJECT_MEMORY_PATH` —— 物体记忆（recall/remember），map 帧坐标可跨运行复用。
- `VLFM_ATTR_VERIFY` / `VLFM_ATTR_PREDICATE` / `VLFM_ATTR_NOUN` —— 二次确认（联网 VLM）。
- `VLFM_REINSPECT` 系列 —— frontier 耗尽时 value-map 引导回看。
- `VLFM_GOAL_SEQUENCE` —— 多目标队列（可选）。

**新增（本系列产物）**
- ObstacleMap 新方法 `update_from_occupancy_grid`（M1）。
- Limo 插线层 `build_observation`（M2）。
- 决策包装 `reset_episode` / `decide_goal` / `on_goal_reached` / `on_goal_unreachable`（M3/M4）。

## 5. 依赖服务（必须先起）

VLFM 大脑是这些推理服务的**客户端**（端口见 `base_objectnav_policy.py` / `itm_policy.py` 默认值）：

| 服务 | 用途 | 默认端口 env |
|---|---|---|
| GroundingDINO | 非 COCO 目标检测 | `GROUNDING_DINO_PORT=12181` |
| YOLOv7（或 YOLO26 TRT） | COCO 目标检测 | `YOLOV7_PORT=12184` |
| MobileSAM | 分割取点云 | `SAM_PORT=12183` |
| ITM（SigLIP2/BLIP2） | value map 语义打分 | `BLIP2ITM_PORT=12182` |
| AttributeVerifier | 二次确认（联网 VLM） | `ATTR_VERIFIER_PORT=12186` |

> 起服务参考 `scripts/dgx`，日志在 `logs/dgx_vlm`。G0/G1 至少要 ITM+YOLO/GDINO+SAM；G2 起再加 AttributeVerifier。

## 6. 怎么验收这套文档「做对了」

每篇模块文档结尾的「验收标准」都写成**可观察 / 可跑命令**的形式（RViz 现象、`ros2 topic echo`、单测断言、日志关键字）。验收时：
1. 按 Gate 顺序，**逐 Gate 卡**，不跳。
2. 每个模块先跑它自带的「离线单测验收」（不依赖机器人），再跑「在线验收」。
3. 任何一条不达标 → 回对应文档「可能问题+解法」表对症，不要往下走。
