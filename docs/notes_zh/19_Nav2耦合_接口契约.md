# 19 · 接口契约（坐标律 + ROS 接口 + Python API）

> 这是全系列的**单一事实来源**。M1–M5 都遵守此契约；改契约必须回来改这里并通知所有模块。

## A. 坐标律（最高优先级，违反必出 bug）

1. **唯一交换坐标 = `map` 帧的米 (x, y)。** VLFM 内部所有图（`ObstacleMap`/`ValueMap`/`ObjectPointCloudMap`，都继承 `BaseMap`）以 (x,y) 米互通；frontier 从 obstacle map 出来是 (x,y)，到 value map 按 (x,y) 查分。
2. **episodic 帧 ≡ ROS `map` 帧。** episodic 原点 (0,0) 就是 `map` 帧原点。喂给 VLFM 的机器人/相机位姿 = `TF map→base_link`／`TF map→camera_optical_frame` 的**原值**。
3. **绝不做 y 取反。** 抛弃 `habitat_policies.py:199` 的 `[x,-y]`（那是 Habitat GPS 约定）。喂进去和读出来都用 map 帧原值。
4. **分辨率对齐：`pixels_per_meter=20` ⇔ `0.05 m/cell`。** 配 RTAB-Map `grid/cellSize=0.05`，`/map` 注入退化为「平移+拷贝」，零重采样误差。
5. **网格尺寸 `size`（像素）= 视场覆盖。** 默认 1000（50 m×50 m，居中于 map 原点）。环境更大或机器人离 map 原点远 → 调大 `size`（如 1600），否则触发 `base_objectnav_policy.py:259` 的 `IndexError: Reached edge of map`。

`BaseMap` 的像素映射（`vlfm/mapping/base_map.py:35-60`，**不要改**）：
```
_xy_to_px:  row = H - (y*ppm + origin) ;  col = x*ppm + origin    # origin = size//2
_px_to_xy:  反变换
```
> 含义：+x→列增（右），+y→行减（上），原点居中。obstacle map 与 value map 内部各自的像素约定不同，但**都以 (x,y) 米为入口**，故 (x,y) 一致即对齐——这是原版不变式，沿用。

## B. ROS 接口表

> topic/frame 名按你实际系统改；用 `ros2 topic list`、`ros2 run tf2_tools view_frames` 先核对。下表给默认值与 ROS 参数名。

### 输入（VLFM 节点订阅）

| 数据 | 默认 topic / frame | 类型 | 用途 | 参数名 |
|---|---|---|---|---|
| RGB | `/camera/color/image_raw` | `sensor_msgs/Image` | value/object map | `rgb_topic` |
| 深度 | `/camera/aligned_depth_to_color/image_raw` | `sensor_msgs/Image`（`16UC1` mm 或 `32FC1` m） | value/object map | `depth_topic` |
| 相机内参 | `/camera/color/camera_info` | `sensor_msgs/CameraInfo` | 取 `fx=K[0]`,`fy=K[4]` | `camera_info_topic` |
| 占据图 | `/map` | `nav_msgs/OccupancyGrid` | 注入 ObstacleMap（M1） | `map_topic` |
| 位姿 | TF `map→base_link` | TF2 | `robot_xy`,`robot_heading` | `map_frame`,`base_frame` |
| 相机位姿 | TF `map→camera_color_optical_frame` | TF2 | `tf_camera_to_episodic` | `camera_frame` |

### 输出（VLFM 节点发布/调用）

| 数据 | 接口 | 类型 | 说明 |
|---|---|---|---|
| **目标** | action `navigate_to_pose` | `nav2_msgs/action/NavigateToPose` | ★唯一目标流，经 `nav2_simple_commander.BasicNavigator` |
| 初始扫描(可选) | action `spin` | `nav2_msgs/action/Spin` | `BasicNavigator.spin()` 转一圈建初始 value map（替代 Habitat `_initialize`） |
| 可视化(可选) | `/vlfm/value_map` | `sensor_msgs/Image` | 调试 |
| 可视化(可选) | `/vlfm/frontiers`,`/vlfm/goal` | `MarkerArray`,`PoseStamped` | RViz 看选点 |

## C. observation cache 结构（VLFM 大脑唯一输入）

M2 的 `build_observation()` 产出、M3 的 `decide_goal()` 消费。键与 `habitat_policies.py:222` 完全一致：

```python
obs_cache = {
    "frontier_sensor": np.ndarray,   # (N,2) map 帧 frontier；来自 M1。无则 np.array([])
    "robot_xy":        np.ndarray,   # (2,)  map 帧，TF map→base_link 原值
    "robot_heading":   float,        # rad，map 帧 yaw
    "nav_depth":       np.ndarray,   # 归一化深度[0,1]；方案B里 Nav2 开车，可填占位
    "object_map_rgbd": [(rgb, depth, tf_cam2map, min_depth, max_depth, fx, fy)],
    "value_map_rgbd":  [(rgb, depth, tf_cam2map, min_depth, max_depth, fov)],  # fov=get_fov(fx,W) 弧度
}
```
约定：
- `rgb`：HxWx3 uint8 RGB。`depth`：HxW float，**已归一化到 [0,1]**（`(d_m-min)/(max-min)`，无效=0 或 1，见 M2）。
- `tf_cam2map`：4x4，`xyz_yaw_to_tf_matrix(cam_xyz_map, cam_yaw_map)`，即 `TF map→camera_optical_frame`。
- `min_depth`/`max_depth`：相机量程（米），与归一化口径一致。
- `fov`：value_map 用，弧度，`get_fov(fx, width)`。

## E. 契约级注意点（跨模块）

- **不要并存两个写者**：A* (`_navigate_global`)、`_pointnav` 在方案 B 全程不调用；它们仍存在于基类，只是不触达。
- **同 size/同 ppm/同原点**：`ObstacleMap` 与 `ITMPolicy` 内部的 `ValueMap` 必须同参数（默认都 size=1000,ppm=20）。M3 构造时核对。
- **深度口径一处定义**：`min_depth`/`max_depth` 在 M2 定义后，M1（如用 depth 投影，本方案不用）、M3、归一化全部引用同一组，禁止各写各的。
- **frame 名核对优先**：任何「机器人往反方向跑 / frontier 错位」先查 frame 名与 y 取反，再查别处（90% 是坐标）。
